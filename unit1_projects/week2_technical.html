Classes and IdsMay 4, 2014
Best Practices for Classes and Ids

Both the class and id attributes are deceptively simple. They are among the easiest aspects of HTML and CSS to implement, and their self-explanatory names make them approachable to even the most frustrated learner. But they are logically embedded into the structure of HTML as much as any other component, and only a good understanding of this logic behind them will allow for optimal utilization.

To understand classes and ids, you must first picture the Document Object Model. You don't need to know the inner workings of the DOM, but you should understand that every element is an object within the Document Object Model, a node on a tree. More specifically, every element is an HTMLElement object. Think of HTMLElements like people on a family tree, unique individuals, but all connected in a hierarchal fashion.

Every HTMLElement object can have properties. A property like a name. And, for simplicity's sake, HTML attributes are just properties by a different name. They are HTTP attributes that are just repurposed when defined in connection to an HTML object so that they can fit nicely into the property definition.

So ids and classes are properties. But what's the difference then, you ask? As you probably know, and id is unique. It is a key. It is an identifier. It is good practice, and commonly understood, to exist singularly. An object can only have one id and that id only exists for that object. If you read otherwise, read until you fully understand what they're saying, but until then take what I say as truth!

But what about the class? You've read about them. An element can have more then one class. How does they gritty HTMLElement object sort this out differently than it does an id? First off, an HTMLELement actually does not have a property called class, it has one called classList. This is, listen closely now, a read-only list of space separated class names. If you know any basic coding, it is like an array. That is how you can fit so many class names into one property. This is why they can be put on numerous elements. There is room for the infinite in the list. And they are just regular words, not keys.

So from knowing that, you can start developing best practices when it comes to using id and classes. Think about how they fit into the HTMLElement before you start sticking them on everything. This will go a long way towards tackling common sense problems. Like, maybe you're wondering if an element can have a class and an id. What about multiple classes and an id? Of course! Think about it. They exist as different properties. They don't tangle up. The answer reveals itself.

But once you start working them, it's important to realize some things. First, they are namers more than anything else. They don't do anything to your elements by default. They allow you to do things to your elements. So use them when you want better control over your elements. They allow for easier selection.

Second, don't use them just to use them. The thing you learn about CSS is that simplicity makes things easier. Classes and ids make things easier. Using classes and ids simply and only when there is need makes things easier! Too many classes or too many ids makes everything bulky. If want to place the same style across multiple elements, stick it on their class rather than any id. You then only write it out once in your CSS file.

Third, ids actually also allow for some special things, like linking to a particular part of a page via an href attribute. Again, if you think about this, it can only apply to ids. Both ids and classes are namers, and you can't move to multiple places on a page if multiple elements have the same class, so only ids allow for a good standard in giving you want you want.

Fourth, be specific as possibly when naming them. They are namers. They are identifiers. Not descriptors. Think about it.

Last but not least, distinguishing between classes and ids is important when it comes to determining CSS specificity with regard to styling. Simply put, ids are more specific then classes. When marking the styling of an element with a class and an id, the id styling will win out, even if the class styling is further down the CSS page, even if you're specifying by a zillion classes and just one id. So when it comes to styling and you're working with classes and ids, be as specific as it makes sense to be. Utilize the ids if you have them. I like to think of it like this. I style based on classes for the foundation, and the unique alterations for individual elements are made via their ids.
